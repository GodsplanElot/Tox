Create a timestamped PostgreSQL backup

mkdir -p backups
docker compose --env-file backend/.env -f docker-compose.prod.yml exec -T postgres sh -c 'pg_dump -U "$POSTGRES_USER" -d "$POSTGRES_DB"' > "backups/toxicreels_$(date +%F_%H%M%S).sql"
ls -lh backups/

-T disables pseudo-terminal output so redirected SQL is clean. Database variables expand inside the container; their values are not printed by the command itself.

Minimum backup validation

test -s backups/<BACKUP_FILE.sql> && echo "Backup file is non-empty"
head -n 5 backups/<BACKUP_FILE.sql>
Never commit a backup to Git. A backup stored only on the same Lightsail instance is not disaster recovery; copy approved backups to protected storage under your retention policy.

Migration rules

  • Deploy committed, reviewed migration files only.
  • Never run makemigrations on production.
  • Back up before schema or production data changes.
  • Do not reverse a migration without reviewing data impact.

7. Environment, Compose, Nginx, and disk commands

Environment-only update

# Edit backend/.env with a secure server editor.
docker compose --env-file backend/.env -f docker-compose.prod.yml config --quiet
docker compose --env-file backend/.env -f docker-compose.prod.yml up -d --force-recreate web

An env change normally needs container recreation, not an image build. Build as well if code, requirements, or the Dockerfile changed.

Production Compose update

git pull --ff-only origin main
git diff HEAD~1 -- docker-compose.prod.yml
docker compose --env-file backend/.env -f docker-compose.prod.yml config --quiet
docker compose --env-file backend/.env -f docker-compose.prod.yml up -d --build

Review service names, bind mounts, named volumes, ports, and PostgreSQL settings before applying a Compose change.

Nginx-only update

sudo nginx -t
sudo systemctl reload nginx
sudo systemctl status nginx --no-pager

Reload only after nginx -t succeeds. Nginx-only changes do not require a Docker rebuild.

Run a reviewed Django command

docker compose --env-file backend/.env -f docker-compose.prod.yml exec web python manage.py <command>

8. Verify from container to public HTTPS

Container and commit checks

docker compose --env-file backend/.env -f docker-compose.prod.yml ps
docker compose --env-file backend/.env -f docker-compose.prod.yml logs --tail=150 web
git rev-parse --short HEAD

Local Gunicorn and public endpoint checks

curl -sS -o /dev/null -w '%{http_code}\n' -H 'Host: api.toxicreels.com' http://127.0.0.1:8000/admin/
curl -sS -o /dev/null -w '%{http_code}\n' https://api.toxicreels.com/admin/

The explicit Host header matters because Django production allowed-host rules may reject a direct 127.0.0.1 request. A 200, 301, or 302 is normally acceptable for the admin path.

What clean verification looks like

  • Both services are Up; PostgreSQL reports healthy.
  • Gunicorn starts workers without repeated exits.
  • No import, database, S3, allowed-host, or missing-env errors appear.
  • The checked-out SHA equals the intended release.
  • Local and public HTTP checks return valid status codes.
  • The exact workflow changed in development works in production.

Diagnostic commands, in order

docker compose --env-file backend/.env -f docker-compose.prod.yml logs --tail=200 web
docker compose --env-file backend/.env -f docker-compose.prod.yml logs --tail=200 postgres
docker compose --env-file backend/.env -f docker-compose.prod.yml exec web python manage.py check
docker compose --env-file backend/.env -f docker-compose.prod.yml exec web python manage.py showmigrations
df -h
docker system df
sudo nginx -t
sudo journalctl -u nginx --since "15 minutes ago" --no-pager
Follow the boundary: container failure → Docker/web logs; local HTTP works but public fails → Nginx/Cloudflare/TLS; database unhealthy → PostgreSQL logs and disk state.
