Deployment Cheat Sheet

A plain-English AWS Lightsail runbook organized by development change type

Purpose

Use this guide after reviewed development changes have been merged and pushed to main. Start with the decision table, run only the actions needed for that release, and always finish with verification.

Golden rule: Pull first. Rebuild when application code or dependencies changed. Migrate when models changed. Collect static files when Django admin/static assets changed. Then recreate the backend and verify it.

TOXICREELSSAFE DEPLOYMENT ORDER

2. Safe default deployment order

#CommandWhat it does
1git status -sbChecks branch and detects server changes.
2git switch mainSelects the production branch.
3git pull --ff-only origin mainDownloads reviewed code; refuses an accidental merge.
4docker compose --env-file backend/.env -f docker-compose.prod.yml build webBuilds a new Django image. Does not start it.
5docker compose --env-file backend/.env -f docker-compose.prod.yml up -d postgresEnsures PostgreSQL runs in the background.
6docker compose --env-file backend/.env -f docker-compose.prod.yml run --rm web python manage.py migrateRuns migrations in a temporary web container.
7docker compose --env-file backend/.env -f docker-compose.prod.yml run --rm web python manage.py collectstatic --noinputCopies admin/static assets when needed.
8docker compose --env-file backend/.env -f docker-compose.prod.yml up -d webCreates/replaces web with the new image.
9docker compose --env-file backend/.env -f docker-compose.prod.yml psDisplays status and database health.

Routine shortcut

docker compose --env-file backend/.env -f docker-compose.prod.yml up -d --build web

This builds and starts web in one command. Use the longer sequence when migrations, static collection, or troubleshooting require explicit control.

3. Docker Compose command dictionary

Command partPlain-English meaning
--env-file backend/.envLoads production substitutions used by Compose, including PostgreSQL values. Keep it on production commands; never print the file.
-f docker-compose.prod.ymlSelects production configuration, not local development.
config --quietValidates Compose and environment substitutions without printing resolved values.

build --no-cache webRebuilds every layer. Slower; use for suspected stale cache.
up -d webStarts/recreates web. -d means detached.
up -d --build webBuilds and starts web in one step.
up -d --force-recreate webRecreates web even when the image is unchanged; useful after environment edits.

psShows service status, ports, and health.
logs --tail=100 webShows the latest 100 backend log lines.
logs -f webFollows live logs; Ctrl+C only stops viewing.

TOXICREELSVERIFY AND RECOVER

5. Verify and recover

Verification commands

docker compose --env-file backend/.env -f docker-compose.prod.yml ps
docker compose --env-file backend/.env -f docker-compose.prod.yml logs --tail=100 web
curl -sS -o /dev/null -w "%{http_code}\n" -H "Host: api.toxicreels.com" http://127.0.0.1:8000/admin/
  • Both production containers should be Up; PostgreSQL should be healthy.
  • Logs should have no import errors, database failures, or repeated worker crashes.
  • The admin check should return 200 or a redirect such as 301/302.
  • Test the exact workflow changed by the release.

Troubleshooting order

  1. Read logs: docker compose --env-file backend/.env -f docker-compose.prod.yml logs --tail=200 web.
  2. Confirm deployed commit: git log -1 --oneline.
  3. Confirm required env variables exist without printing secret values.
  4. Never use down -v or reverse migrations without a reviewed plan.

Code rollback

git log --oneline -10
git switch --detach <known-good-commit>
docker compose --env-file backend/.env -f docker-compose.prod.yml up -d --build web

TOXICREELSPRE-DEPLOYMENT AND BACKUP

6. Pre-deployment gate and database safety

Connect, enter the repository, and inspect

cd <APP_DIRECTORY_ON_LIGHTSAIL>
git status -sb
git branch --show-current
git fetch origin
git log --oneline --decorate -5
docker --version
docker compose version
docker compose --env-file backend/.env -f docker-compose.prod.yml config --quiet
docker compose --env-file backend/.env -f docker-compose.prod.yml ps
GateExpected result
Git stateBranch is main; there are no unexplained server-side edits.
Remote releaseThe intended commit exists on origin/main. Lightsail cannot pull a commit that only exists locally.
Compose validationconfig --quiet returns no error and does not expose resolved values.
Service baselineWeb is Up; PostgreSQL is Up and healthy before deployment begins.

Create a timestamped PostgreSQL backup

mkdir -p backups
docker compose --env-file backend/.env -f docker-compose.prod.yml exec -T postgres sh -c 'pg_dump -U "$POSTGRES_USER" -d "$POSTGRES_DB"' > "backups/toxicreels_<TIMESTAMP>.sql"
ls -lh backups/

-T disables pseudo-terminal output so redirected SQL is clean. Database variables expand inside the container; their values are not printed by the command itself. Replace <TIMESTAMP> with a date-time label.

Minimum backup validation

test -s backups/<BACKUP_FILE.sql> && echo "Backup file is non-empty"
head -n 5 backups/<BACKUP_FILE.sql>
Never commit a backup to Git. A backup stored only on the same Lightsail instance is not disaster recovery; copy approved backups to protected storage under your retention policy.

Migration rules

  • Deploy committed, reviewed migration files only.
  • Never run makemigrations on production.
  • Back up before schema or production data changes.
  • Do not reverse a migration without reviewing data impact.

7. Environment, Compose, Nginx, and disk commands

Environment-only update

# Edit backend/.env with a secure server editor.
docker compose --env-file backend/.env -f docker-compose.prod.yml config --quiet
docker compose --env-file backend/.env -f docker-compose.prod.yml up -d --force-recreate web
docker compose --env-file backend/.env -f docker-compose.prod.yml logs --tail=100 web

An env change normally needs container recreation, not an image build. Build as well if code, requirements, or the Dockerfile changed.

Production Compose update

git pull --ff-only origin main
git diff HEAD~1 -- docker-compose.prod.yml
docker compose --env-file backend/.env -f docker-compose.prod.yml config --quiet
docker compose --env-file backend/.env -f docker-compose.prod.yml up -d --build

Review service names, bind mounts, named volumes, ports, and PostgreSQL settings before applying a Compose change.

Nginx-only update

sudo nginx -t
sudo systemctl reload nginx
sudo systemctl status nginx --no-pager

Reload only after nginx -t succeeds. Nginx-only changes do not require a Docker rebuild.

Run a reviewed Django command

docker compose --env-file backend/.env -f docker-compose.prod.yml exec web python manage.py <command>

Inspect disk usage and reclaim dangling images

df -h
docker system df
docker image ls
docker image prune -f
CommandProduction meaning
image prune -fRemoves dangling images only. Run after the new release is verified.

volume pruneCan remove unused volumes. Avoid unless every candidate volume is reviewed.

TOXICREELSDEEP VERIFICATION

8. Verify from container to public HTTPS

Container and commit checks

docker compose --env-file backend/.env -f docker-compose.prod.yml ps
docker compose --env-file backend/.env -f docker-compose.prod.yml logs --tail=150 web
git rev-parse --short HEAD

Local Gunicorn and public endpoint checks

curl -sS -o /dev/null -w '%{http_code}\n' -H 'Host: api.toxicreels.com' http://127.0.0.1:8000/admin/
curl -sS -o /dev/null -w '%{http_code}\n' https://api.toxicreels.com/admin/

The explicit Host header matters because Django production allowed-host rules may reject a direct 127.0.0.1 request. A 200, 301, or 302 is normally acceptable for the admin path.

What clean verification looks like

  • Both services are Up; PostgreSQL reports healthy.
  • Gunicorn starts workers without repeated exits.
  • No import, database, S3, allowed-host, or missing-env errors appear.
  • The checked-out SHA equals the intended release.
  • Local and public HTTP checks return valid status codes.
  • The exact workflow changed in development works in production.

Diagnostic commands, in order

docker compose --env-file backend/.env -f docker-compose.prod.yml logs --tail=200 web
docker compose --env-file backend/.env -f docker-compose.prod.yml logs --tail=200 postgres
docker compose --env-file backend/.env -f docker-compose.prod.yml exec web python manage.py check
docker compose --env-file backend/.env -f docker-compose.prod.yml exec web python manage.py showmigrations
df -h
docker system df
sudo nginx -t
sudo journalctl -u nginx --since "15 minutes ago" --no-pager
Follow the boundary: container failure → Docker/web logs; local HTTP works but public fails → Nginx/Cloudflare/TLS; database unhealthy → PostgreSQL logs and disk state.

9. Lightsail deployment checklist

Before

  • Release is merged into main and pushed to origin/main.
  • Release type is identified from page 2.
  • A database backup exists when migrations or data changes are involved.
  • The server working tree has no unexplained edits.

Deploy

cd <APP_DIRECTORY_ON_LIGHTSAIL>
git status -sb
git switch main
git pull --ff-only origin main
docker compose --env-file backend/.env -f docker-compose.prod.yml config --quiet
docker compose --env-file backend/.env -f docker-compose.prod.yml build web
docker compose --env-file backend/.env -f docker-compose.prod.yml up -d postgres

# Run when migrations exist (also safe as a schema-current check):
docker compose --env-file backend/.env -f docker-compose.prod.yml run --rm web python manage.py migrate

# Run only when Django static/admin assets changed:
docker compose --env-file backend/.env -f docker-compose.prod.yml run --rm web python manage.py collectstatic --noinput

docker compose --env-file backend/.env -f docker-compose.prod.yml up -d web

After

docker compose --env-file backend/.env -f docker-compose.prod.yml ps
docker compose --env-file backend/.env -f docker-compose.prod.yml logs --tail=100 web
git rev-parse --short HEAD
curl -sS -o /dev/null -w '%{http_code}\n' -H 'Host: api.toxicreels.com' http://127.0.0.1:8000/admin/
curl -sS -o /dev/null -w '%{http_code}\n' https://api.toxicreels.com/admin/
Deployment is complete only when services are healthy, logs are clean, the SHA matches, HTTP checks respond correctly, and the changed workflow passes.
